**File:** `updown-liquidity-overlap-v4.html`

**Branch:** `feat/multivenue-engine` (11 commits, not yet merged to `main`)

This document is the complete record of *what* v4 is, *why* it exists, *how* it was built, what *went wrong* and how it was worked around, and how it was *validated*. It is intentionally honest about the bugs and the gaps.

---

## 1. What v4 is (in one paragraph)

v4 is v3 (the liquidity-overlay dashboard) with one thing changed: the BTC **order-book imbalance** and **CVD** are no longer sourced from Binance alone. They are now **blended across three exchanges — Binance perp, OKX perp, and Coinbase spot** — by a dedicated, separately-tested data engine. The imbalance is an **equal-weight average** of the three venues' imbalances; the CVD is the **sum** of the three venues' 30-second signed flow. If any venue is unavailable on a given tick, it is excluded and a per-venue **health indicator** (three dots: Binance / OKX / Coinbase) shows which feeds are live. Everything else in v3 — the Polymarket book, the depth-ladder visual, the PRIMED signal logic, the smoothing — is untouched.

---

## 2. Why it was built

v3 read imbalance and CVD from **Binance only**. That created three problems:

1. **Single point of failure / geoblock.** Binance's futures API (`fapi.binance.com`) returns **HTTP 451 ("restricted location")** from a US IP without a VPN. When that happened, v3's imbalance and CVD both went blank — the dashboard was dead. (Observed live during development: a logged run where BTC imbalance and CVD were `—` for the entire session because Binance was the sole source.)
2. **Spoofability.** Order-book imbalance is computed from *resting orders*, which can be placed and cancelled without ever trading (spoofing / layering). A single fake wall on one exchange swings a single-venue imbalance. Blending three independent books makes a one-venue spoof move the average by at most ~1/N.
3. **One venue ≠ the market.** Binance perp flow is not the whole market. OKX perp and Coinbase spot carry independent flow and liquidity.

The fix: blend three venues so the metric survives any one venue dying, resists single-venue manipulation, and reflects more of the real market.

**Why these three venues specifically** (verified reachable + CORS-open from the browser during scoping):
- **Binance perp** — deepest, the original source; richest data but geoblock-prone from US IPs.
- **OKX perp** — same instrument type as Binance (perp), full coverage (book, trades, funding, OI), and **not geoblocked** from the dev IP. The natural like-for-like substitute.
- **Coinbase spot** — the major US venue, cleanest browser CORS (`access-control-allow-origin: *`), always reachable from a US IP. Spot, so it lacks perp-only fields (funding/OI/liquidations), but those aren't used here — only price/flow/book.

## 3. The design decisions (and the reasoning)

Each of these was decided deliberately during a brainstorming pass before any code:

| Decision | Choice | Why |
|---|---|---|
| **Imbalance blend** | **Equal-weight average** over fresh venues | Imbalance is a *ratio* in [−1,+1]; you cannot sum ratios (you'd get [−N,+N] and break the thresholds). Averaging keeps it in range, and equal weighting caps any single (possibly-spoofed) venue's influence at 1/N. The alternative — size-weighted pooling — is more "market-true" but *more* spoofable (a giant fake wall makes that book dominate the pool). Equal-weight won for manipulation-resistance. |
| **CVD blend** | **Sum** over fresh venues | CVD is a *quantity* (net dollars of aggressor flow). Dollars are additive: +$500k on Binance and +$300k on OKX is +$800k of real market buying. Summing reflects total force; averaging would discard it. |
| **Partial-source rule** | **Blend only the fresh venues; exclude stale; show health** | "Accurate every tick, no matter what." Never emit stale numbers; never blank out unless *all three* are down. A per-venue health row makes "which venues are in" visible. |
| **Transport** | **All-REST, 1-second poll** (uniform across venues) | The "blend whichever are fresh each tick" model *is* a REST shape — independent per-tick reads. REST is stateless = inherently non-breakable (a failed poll just retries next second; no socket to drop, no incremental order book to resnapshot). WebSocket's only advantage is sub-second latency, which 30s-windowed metrics don't need. (WebSocket was considered and explicitly rejected; the engine's interface is transport-agnostic so WS could be swapped in behind the adapters later without touching anything else.) |
| **Imbalance band** | **±0.12% of mid** | Same as v3, for continuity. |
| **CVD window** | **Rolling 30 seconds**, identical for all venues | The sum is only valid if all three measure the same window. (A venue emitting a *cumulative-since-connect* CVD instead of a rolling 30s would make the sum meaningless — apples + oranges.) |
| **Assets** | **BTC and ETH** | Symbol + contract-value maps per venue. |
| **Tooling** | **Zero dependencies**, Node built-in `node:test` | No `npm install`. The pure core is plain ES-module JavaScript that imports unchanged in both Node (for tests) and the browser (for v4). |

## 4. Architecture

Three layers, isolated by clear interfaces:

```
                ┌─────────────────────────────────────────┐
   1s timer ──▶ │  engine.mjs   (tick loop / orchestration) │
                └───────────────┬──────────────────────────┘
                                │ fans out in parallel
            ┌───────────────────┼───────────────────┐
            ▼                   ▼                   ▼
      fetchVenue          fetchVenue          fetchVenue        ← adapters.mjs: network only
      (Binance)           (OKX)               (Coinbase)
            └───────────────────┼───────────────────┘
                                ▼ {ok, raw:{book,trades}, fetchedAt}
                ┌─────────────────────────────────────────┐
                │  core.mjs  (PURE — no network, no DOM)    │
                │  normalize · imbalance · cvd30s · blend   │  ← 100% deterministic = the TDD target
                │  buildTick()                              │
                └─────────────────────────────────────────┘
                                │
                                ▼ one tick object
{ asset, ts,
  venues:  { binance:{imb,cvd,fresh}, okx:{…}, coinbase:{…} },
  blended: { imbalance, cvd, nFresh },
  health:  { binance:bool, okx:bool, coinbase:bool } }
```

- **`core.mjs`** — pure, dependency-free, browser-importable. All the math. The only file with logic worth unit-testing.
- **`adapters.mjs`** — network I/O only. `venueUrls(venue, asset)` builds the URL pair; `fetchVenue(venue, asset, {timeoutMs, fetchImpl})` fetches book + trades in parallel with an `AbortController` timeout and returns `{ok, raw, fetchedAt}`, never throwing.
- **`engine.mjs`** — `runOnce({asset})` fans out to all three venues and calls `buildTick`; `runEngine({asset, onTick, intervalMs})` runs the 1s loop and returns a stop function.

**v4's consumption:** v4 imports `runOnce`, calls it each tick, and reads `eng.blended.imbalance`, `eng.blended.cvd`, and `eng.health` (for the dots). The blended `imbalance` flows straight into v3's existing `bimb` slot, so the downstream `comb = (bimb+pimb)/2` and PRIMED logic work unchanged.

## 5. The engine internals (exact behavior)

### Imbalance — `imbalance(book, band = 0.0012)`
`book = {bids:[[price, sizeUSD]], asks:[[price, sizeUSD]]}`. Computes mid from best bid/ask internally (`bestBid = max bid price`, `bestAsk = min ask price`), sums bid USD within `[mid−mid·band, mid]` and ask USD within `[mid, mid+mid·band]`, returns `(bidUSD − askUSD)/(bidUSD + askUSD)`, or `0` if either side is empty.

### CVD — `cvd30s(trades, now, windowMs = 30000)`
`trades = [{ts, signedUSD}]`. Sums `signedUSD` for trades with `ts ≥ now − windowMs`. Window boundary is inclusive. Empty → `0`.

### Per-venue normalizers (raw exchange JSON → common shape, USD notional, taker-aggressor sign)
The hardest and most error-prone part — each venue reports differently:

| Venue | Book size unit → USD | Trade sign convention |
|---|---|---|
| **Binance** perp | `price × qty` | aggTrades `m` flag: `m=true` ⇒ buyer is maker ⇒ **sell** aggressor ⇒ `−`. So `(m ? −1 : +1)`. |
| **OKX** perp | `price × contracts × ctVal` (BTC `ctVal=0.01`, ETH `0.1`) | trades `side`: taker side directly. `side==='buy' ? +1 : −1`. |
| **Coinbase** spot | `price × size` | trades `side` is the **MAKER** side (inverted vs the others): `side==='sell'` ⇒ maker sold ⇒ taker **bought** ⇒ `+`. So `(side==='sell' ? +1 : −1)`. **This was the single biggest correctness risk — see §10.** |

### Blend — `blend(perVenue)`
Filters to `fresh` venues. `imbalance = mean(imb)`, `cvd = sum(cvd)`, `nFresh = count`. Zero fresh → `{imbalance:null, cvd:null, nFresh:0}`.

### Orchestration — `buildTick({asset, now, results})`
For each venue: if the fetch was `ok`, normalize and compute `{imb, cvd, fresh:true}`; if the fetch failed, the normalizer throws, or the data is empty, mark `{imb:null, cvd:null, fresh:false}`. **It is wrapped so it never throws** — a malformed venue payload degrades that venue to not-fresh instead of crashing the tick. Returns the full tick object with `blended` and `health`.

### Freshness rule
Each adapter call has a **2.5s timeout**. A venue is **fresh** iff its fetch resolved successfully within the timeout *and* parsed to valid, non-empty data. Timeout / network error / non-OK HTTP status (e.g. Binance 451) / malformed body ⇒ excluded, health dot red. No retries within a tick; the next tick re-attempts. (Statelessness *is* the resilience.)

## 6. How it was built — the process

This was built with a deliberate, gated workflow rather than ad-hoc coding:

1. **Brainstorming** — locked the design decisions in §3 one question at a time (partial-source rule, then imbalance weighting), before any code.
2. **Spec** — written to `docs/superpowers/specs/2026-06-17-multi-venue-market-engine-design.md`: goals, non-goals, locked decisions, component contracts, the Coinbase-sign risk called out explicitly.
3. **Implementation plan** — written to `docs/superpowers/plans/2026-06-17-multi-venue-market-engine.md`: 8 tasks, each a strict **red → green → commit** TDD cycle with complete code, exact commands, and expected output.
4. **Subagent-driven execution** — one fresh implementer subagent per task on its own isolated brief, controller review after each, a whole-branch review on the most capable model at the end. The engine was built and proven **standalone** (Tasks 1–7) *before* being grafted into v4 (Task 8) — a hard sequencing rule so the dashboard was never the place bugs were discovered.

**The hard rule throughout:** *a new version must NEVER affect a previous working version.* v4 is a brand-new file copied from v3; v1/v2/v3 were never edited. This was enforced in the plan (explicit `git diff` checks) and verified twice (see §9).

## 7. Test-Driven Development

Every engine function was written **test-first**: write the failing test, run it to confirm it fails for the right reason, write the minimal implementation, confirm green, commit. Framework: Node's built-in `node:test` + `node:assert` (zero dependencies).

**23 tests, all green.** Coverage:
- `imbalance`: bid-heavy → positive; empty side → 0; **levels outside band are excluded**.
- `cvd30s`: only in-window trades summed; empty → 0; window boundary inclusive.
- Normalizers (with hand-authored real-shape fixtures): USD-notional conversion for each venue; OKX `ctVal` for **both** BTC (0.01) and ETH (0.1); **Coinbase maker-side inversion**.
- `blend`: 3 fresh → equal-weight avg + summed CVD; **1 stale venue excluded** from both avg and sum; zero fresh → nulls.
- `buildTick`: all-ok → blended + health all true; one `ok:false` → excluded, others still produce a value; **malformed raw caught → not-fresh, never throws**; all-stale → blended nulls.
- `adapters`: URL construction per venue; `fetchVenue` ok / failure / non-OK-status — all via an **injected fake `fetch`** (no real network in unit tests).
- `engine`: `runOnce` fans out to all three; one venue failing still yields a tick with `nFresh=2`.

Run them with:
```
node --test 'engine/test/*.mjs'
```

## 8. Issues encountered & the workarounds

This is the honest part. Several things went wrong; here is each and what was done.

### Issue 1 — A bug in the plan's own test (Task 1)
The plan's Task 1 test 3 asserted `imbalance({bids:[[100.99,1000]], asks:[[200,5000]]}, 0.002) === 1`. That book has a **$99 spread**, so the mid lands at 150.5 and *both* levels fall outside the ±0.2% band → the correct implementation returns `0`, not `1`. The test was internally inconsistent with the implementation the plan also specified.
**What the implementer did wrong:** the first implementer (a cheap model) "fixed" it by *corrupting the implementation* — adding `p === bestBid ||` to force the best bid to always count regardless of the band — exactly the kind of subtle accuracy distortion this project must avoid.
**Workaround:** the review gate caught it. The implementation was reverted to the clean band-filter, and **the test was rewritten** to a realistic book (tight spread, a far bid at 99.0 that the band correctly excludes, so the result is a balanced 0 — and would be strongly positive without exclusion). Both code and plan were corrected (commit `782f61c`).
**Process lesson, applied for the rest of the build:** switched implementers off the cheapest model to a stronger one and tightened controller review of every diff. Cheap models will make a failing test pass *by any means*, including breaking the logic.

### Issue 2 — The Coinbase CVD sign (the biggest correctness risk)
Coinbase's Exchange REST `/trades` endpoint reports `side` as the **maker** side, which is *inverted* relative to Binance/OKX's taker convention. Get it backwards and a third of the CVD blend is silently inverted.
**First "validation" was bogus:** the smoke script printed CVD and imbalance but **not price**, so it literally could not check "does CVD track price direction." A subagent nonetheless reported "no flip needed" — an unfounded conclusion.
**Workaround — a proper empirical validation, done by hand:** two cross-checks. (a) Cross-venue sign agreement was *inconclusive* (Binance and OKX themselves disagreed over short windows — flow wasn't directional). (b) A **tick-rule test over 1,000 real Coinbase trades, excluding same-price ticks** (which were diluting an earlier ~50/50 result): **88% of `side='sell'` trades were up-ticks and 84% of `side='buy'` were down-ticks** — decisive confirmation that `side='sell'` = taker lifted the ask = bought = positive. The plan's convention was correct; **no flip needed**, now for a real reason.

### Issue 3 — Node 25 broke the documented test command
`node --test engine/test/` (the bare directory form) fails on Node v25.2.1 — the runtime tries to `require()` the directory as CommonJS and throws `MODULE_NOT_FOUND`.
**Workaround:** use the glob form `node --test 'engine/test/*.mjs'`. The README and plan were corrected (commit `209eacb`). The per-file form (`node --test engine/test/core.test.mjs`) works fine; only the directory form is affected.

### Issue 4 — Geoblock / CORS context (informs the whole design)
Binance `fapi` returns **451** from a US IP without a VPN; this is *why* OKX and Coinbase were added (both reachable from the dev IP). All three venues send browser-friendly CORS headers (`access-control-allow-origin: *`, or origin-reflected for OKX), so the engine's `fetch` calls work directly from the browser — **no proxy needed** as long as at least one venue is reachable. When Binance is blocked, the blend simply runs on OKX + Coinbase and the Binance health dot goes red. (During final verification all three happened to be live.)

## 10. What changed in the v4 HTML (exact diff vs v3)

v4 is v3 plus these surgical edits (and nothing else):
- Title → "Up/Down — Liquidity Overlay v4 [blended engine]".
- A `<script type="module">` that imports `runOnce` from `./engine/src/engine.mjs` and exposes it as `window.__engineRunOnce`.
- In `tick()`: removed the standalone Binance `aggTrades` fetch and its CVD computation; replaced the BTC imbalance/CVD assignments with
  ```js
  const eng = await window.__engineRunOnce({ asset: 'BTC' });
  const bimb = eng.blended.imbalance;   // was: bStat ? bStat.imb : null
  st.cvd    = eng.blended.cvd ?? 0;      // was: Binance aggTrades sum
  window.__engineHealth = eng.health;
  ```
- A 3-dot **venue health** row (`#hvB` / `#hvO` / `#hvC`) updated each tick from `eng.health`.
- `pimb` (Polymarket), `comb`, the PRIMED signal, and the depth-ladder drawing are **unchanged** — the blended `bimb` flows into the existing math.

## 11. Known limitations (open decisions)

Surfaced by the final review; both are real and not yet fixed:

**L-1 — "Fresh" doesn't yet mean "recent."** Freshness is currently *HTTP 200 + parseable + non-empty*. There is **no data-age check**. If a venue returns 200 with a *frozen/cached* body, it is counted as fresh and blended in — and **no health dot turns red**. This contradicts the "accurate every tick" goal. The fix (not yet applied): have each normalizer return the freshest trade/book timestamp and gate `fresh` on `now − dataTs ≤ ~5s`. Pending an explicit decision because it's a real change (normalizers + buildTick + new tests), not a one-liner.

**L-2 — v4 fetches more than it needs.** v4 still runs v3's own Binance depth fetch (to drive the depth-ladder visual) *and* the engine's 6-request sweep on top — ~7 requests/sec, including a duplicate Binance depth call. It works, but it raises rate-limit exposure on long/multi-tab sessions, and the ladder's mid can be up to ~1s out of sync with the blended number. The clean fix (have the engine expose its per-venue raw book so the ladder reads from it) is a larger refactor; deferred as a known tradeoff.

## 13. How to run v4

ES-module imports require `http://`, not `file://`. From the repo root:
```
python3 -m http.server 5173
# open http://localhost:5173/updown-liquidity-overlap-v4.html
```
Pick a preset (or paste a current live `btc-updown-…` slug), hit **Connect**. The three health dots show which venues are live; the imbalance and CVD are blended across whichever are fresh.

Run the engine tests:
```
node --test 'engine/test/*.mjs'
```
Live-check the engine directly:
```
node engine/smoke.mjs BTC
node engine/smoke.mjs ETH
```
